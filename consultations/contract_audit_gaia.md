This is a spec audit of the contract text (lines 796–846), not a code audit — so several holes below are *conditional on environment facts the contract leaves unpinned* (a11y bus topology, Send↔Stop toggling, whether uploads/follow-ups are in scope, X vs Wayland for the file chooser). Where I can't verify from the text I mark it **[Inferred]**. The fact that these are unpinned is itself a finding.

## The keystone finding (everything else hangs off this)

The contract asserts **"drift is the only unknown"** (L823) and routes every no-match through one pipe: settle → rescan → notify-as-drift. But a no-match has **three** distinct causes, and the contract collapses all three into the third:

1. **Timing** — tree not refreshed yet (the contract handles this).
2. **Drift** — a mapped element genuinely changed name/role (the contract handles this).
3. **Alternate legitimate state** — the UI forked to a *different, correctly-rendered* screen the map doesn't enumerate (auth wall, rate-limit modal, mid-gen error, "continue generating", A/B chooser, policy banner). **The map didn't go stale; it was never complete.**

Calling (3) "drift" is wrong on two axes: it destroys the diagnostic (operator gets "expected Stop, found [Upgrade-modal nodes]" and must reverse-engineer "oh, rate-limited"), and — when combined with the weak completion signal below — it lets garbage land as a real answer. **"Match-or-notify" is sound; "match = the one expected element, and any miss = drift" is not.** The fix recurs throughout: widen "match" from *one expected element* to *exactly one of an enumerated set of mapped states*, each carrying a deterministic disposition. That stays fully binary (one-of-mapped vs none) and fully no-guess.

## Prioritized holes + amendments

**Tier 0 — can produce a *silent wrong result* (violates the no-guess guarantee invisibly; worst class):**

**P1 [CONTRADICTION].** *"Drift is the only unknown" (L823) / "the map is finite, known, and complete" (L813).* Untrue against real platforms: alternate states aren't drift. **Amend:** YAML maps, per decision point, the **full reachable state set** (happy-path + every alternate). A step "matches" iff the windowed tree satisfies **exactly one** mapped state. Each non-happy state carries an explicit disposition: `HALT+notify(reason_code)` (auth-wall, rate-limit, captcha, length-error, policy-block) or a mapped recovery action that is itself match-or-notify (continue-generating). Replace "no-match ⇒ drift" with "no-match ⇒ none-of-the-mapped-set ⇒ notify(unmapped-state, candidates)." Drift becomes one reason code among several.

**P2 [CONTRADICTION/GAP].** *"Nothing is ever hidden… never invisible-but-present" (L836–837)* is false as an AT-SPI axiom — occluded-by-modal nodes, `aria-hidden` subtrees, lazily-computed Chromium nodes (only populated on focus), offscreen virtualized rows are all present-but-not-actionable. The danger: the composer node still matches name+role **behind a blocking modal**, so the engine types your prompt into an occluded field. **Amend:** (a) snapshot scope = the **whole top-level window incl. portal roots**, never the chat subtree, so overlays are always visible. (b) Add a `MODAL_PRESENT` guard (dialog role at root with `MODAL` state): if present and unmapped → notify; a mapped modal state wins over any underlying happy-path element. (c) Demote the axiom to an *enforced predicate*: the engine only acts on `SHOWING ∧ VISIBLE ∧ SENSITIVE` nodes and never assumes a visually-present-but-tree-absent element exists.

**P3 [GAP, Inferred].** *Cross-instance ambiguity.* Five identical browsers each expose an identical "Send"/"Stop". If they share an AT-SPI registry/session bus, a driver scoped to "the browser app" can confidently match the **wrong display's** element. Exact name+role does **not** prevent this. **Amend:** State the isolation invariant — each browser runs against an isolated a11y bus (`AT_SPI_BUS`/`DBUS_SESSION_BUS_ADDRESS` per display) so tree scope = exactly one window. If the bus is shared, the YAML locator **must** include a per-instance discriminator (display/PID/window-title) in the match path, and exact name+role alone is declared insufficient.

**P4 [GAP/CONTRADICTION].** *Completion = "Stop gone" (L832)* is a single **negative** signal with multiple false-success modes: error-mid-generation (Stop vanishes, answer is an error string), "Continue generating" (Stop vanishes, answer is truncated), error→Retry, or a transient unmount mid-stream. Absence is also exactly what a stale snapshot looks like. **Amend:** Completion = a **positive "completed" state** — terminal action bar (copy/regenerate) present **AND** composer re-enabled **AND** Stop absent — and it must be *distinguished* from mapped {truncated-continue, error-retry, A/B-chooser, policy-refusal} states, each with its own disposition. Require the completed-state to hold across the settle window, not a single absence read.

**P5 [GAP/CONTRADICTION].** *Submit success = new-chat URL + Stop appeared (L831)* races (URL and Stop aren't atomic), false-negatives on follow-ups-in-existing-chat (no new URL by construction) and on pushState routing AT-SPI may not surface, and false-positives if the platform uses a **Send↔Stop toggle on one node** (then "Stop present" can be the same node as "Send present" with a changed name) or if Stop is present-but-`insensitive` during spin-up. **Amend:** Define submit-success as the **"generating" state matched** (Stop present *with required states* + composer cleared/disabled), not two booleans. Map Send↔Stop explicitly as one node with two named+stated variants; require the *Stop* name+role+`SENSITIVE` exactly. State the precondition **"engine only initiates new chats"** — or add a second success signal for follow-ups (composer cleared + Stop present + assistant-turn count incremented).

**P6 [CONTRADICTION/GAP].** *"Exact name+role" (L803) under-defines cardinality.* Multiple "Copy"/"Regenerate"/"Send" instances mean name+role isn't unique. Silent `.first()` is a guess. **Amend:** **Match := exactly one** node satisfying the locator within a **mapped scope/containment path**. `0 matches → notify(drift)`; `≥2 → notify(ambiguous)` (distinct reason). The YAML element spec gains an ancestor/container path sufficient for uniqueness.

**P7 [CONTRADICTION].** *Match ignores StateSet.* A disabled Send, a greyed plan-locked model option, an offscreen-but-mounted action bar all match name+role but aren't actionable. **Amend:** Match includes required **StateSet predicates** (`SHOWING ∧ VISIBLE ∧ SENSITIVE ∧ ENABLED`, `FOCUSABLE` for inputs). "Present but disabled" then correctly reads as no-match-for-this-step → settle (it may enable) → notify if still disabled. Add a deterministic `ensure_visible`/`scroll_into_view` primitive so offscreen-but-mounted elements (`SHOWING=false`) are brought on-screen before matching rather than spuriously notifying. **Note: the contract's identity is "name+role"; it must become "stable-locator + role + required-states."**

**P8 [GAP].** *Locating by visible `name` — the least stable AT-SPI property.* The model-picker trigger's name **is** the selected model, so it changes after every pick → next run notifies (or, if mapped loosely to dodge that, selects the **wrong model** — a correctness fault, not just noise). Names with counters/timestamps ("Stop (3)") can't be exact-matched without the banned substring ops. Locale change wipes every English name at once. **Amend:** Mandate locating controls by **stable handles** — `automationId`/`data-testid`/object-attributes (still *exact* match, just on a stable key) or role+container-path — and **never assert intrinsically-dynamic names**. Bless attribute-exact-match as a first-class locator in the contract. Pin locale as a hard precondition and state that name-maps are locale-specific artifacts.

**Tier 1 — hangs / unbounded waits (safe-ish, but break "100% reliable"):**

**P9 [CONTRADICTION].** *"Settle once and re-scan; if still no-match → notify" (L805, L827)* uses **one fixed delay for two different magnitudes**: sub-second portal/render lag vs multi-second network-bound transitions (navigation commit, generating-screen appearance, completion). One constant is either too short (premature false-notify on every slow transition) or too slow. **Amend:** Two constants per platform: `settle_ms` (render debounce) and `transition_timeout_ms` (bounded wait for an *expected* transition). Step validation = **poll-until-match up to `transition_timeout_ms`, settling between polls**, then notify. The invariant isn't "exactly one rescan" — it's *bounded-wait-then-notify*. Still match-or-notify.

**P10 [GAP].** *No generation watchdog.* Stream stalls / dead WebSocket → infinite "generating" (no timeout anywhere). **Amend:** Per-platform `generation_timeout`: if neither "completed" nor any mapped terminal state is reached within T → notify(stall).

**P11 [GAP].** *Notify is a single point that can fail silently.* If `claude-code-fleet-notify` is down/rate-limited, HALT happens but no human is told → invisible stall. HALT granularity, blocking-vs-parking, and cross-platform dedupe are unspecified. **Amend:** Notify must be **delivery-acknowledged**; unacked notify is itself an error state, locally logged + retried + surfaced on a secondary channel. Define HALT as **step-level** (run continues for other platforms), parked in a queryable "needs-human" state so a dead channel can't hide it. Dedupe identical drift across the 5 platforms.

**P12 [CANT-EXPRESS→primitive gap].** *Hover-to-expand flyouts* can't be opened by the listed primitives (`click, click-react, paste, snapshot, menu_snapshot, settle` — L815): no `hover`/`pointer_move`. A hover-triggered submenu is then a **dead-end** — match-or-notify with **no reachable match** (notify forever). Plus hover menus dismiss on mouse-leave (the rescan can collapse the menu) and may render as **root-portal siblings**, not menu children, so scoped `menu_snapshot` misses them. **Amend:** Add `hover`/`pointer_move` and, where exposed, `menu_key` (arrow/access-key nav is deterministic and dodges the mouse-leave race — prefer it). YAML maps each flyout's **trigger type** (click/hover) and **tree attachment point** (child vs root-portal). Define submenu-open validation as a state that holds *while the pointer rests on the trigger*; the rescan re-establishes hover as part of the primitive.

**P13 [GAP/precondition].** *A surface exposes nothing to AT-SPI.* Often a *config* fault — Chromium computes a11y lazily and may be empty unless renderer accessibility is forced on or the node is focused → mass false-notify. **Amend:** Hard launch precondition: browser started with renderer accessibility forced on (mapped as a launch invariant); engine focuses the relevant container before snapshot where nodes are focus-gated. (The *fundamental* canvas case is in the fold-back section.)

**P14 [GAP].** *Streaming tree-thrash.* During generation the message subtree mutates continuously; any match against it is racy, and a momentary unmount can read as "Stop gone." **Amend:** State-detection during generation keys **only** on a mapped low-churn "control strip" (Stop/Send node + composer), explicitly excluding the streaming message subtree; require the completed-state (P4) rather than bare Stop-absence to debounce mid-stream blips.

**Tier 2 — operational correctness:**

**P15 [GAP].** *Re-run double-submit.* The loop notify→fix-YAML→re-run isn't idempotency-guarded. If drift hit *after* submit (e.g., on completion-detection), re-running can **re-send the prompt** — a duplicate turn the no-retry-action ethos (L812) is supposed to forbid, but only forbids *within* a run. **Amend:** Make the run a small state machine with durable checkpoints (`submitted? url? completed?`). Re-run **resumes** from the captured chat URL; submit is idempotency-guarded (URL/turn exists for this run → resume at completion-detection, never replay).

**P16 [GAP/scope].** *File-upload dialog.* The native chooser is a **separate AT-SPI application** (separate top-level), and under Wayland/xdg-desktop-portal it may not be on the a11y bus at all. The contract maps the attach *menu* but never the chooser; "the file tile shows on screen" (L836) is the post-upload chip, not the act of choosing. **Amend:** Decide and state whether the engine uploads. If yes: map the chooser as a distinct AT-SPI app target (location bar/filename/Open) **or** mandate a dialog-free path (synthesized drag-drop onto composer / pre-staged files). If the desktop stack can't expose the chooser (Wayland portal), declare uploads **out of scope** on that platform → notify if requested. (Fold-back, not fallback.)

**P17 [CANT-EXPRESS/honesty].** *"Complete" ≠ "answered."* A clean completion can be a refusal/empty turn; "Stop gone" can't tell. That's acceptable behavior but a spec-honesty gap. **Amend:** State explicitly: **completion = "model finished a turn," content-agnostic**; interpreting content is out of contract scope. (Don't let "complete" imply "useful answer" — your cannot-lie standard.)

## (c) Cases the binary genuinely can't express — and the fold-back

The fold-back principle for almost all of the above: **a "case the binary can't express" is really an *unmapped state* or an *unstable locator* — the fix is a new mapped state / a new validation predicate / a stable handle, never a fuzzy branch.** P1, P4, P5, P12, P22-class all fold back this way.

**The one honest exclusion (not a fold-back, not a fallback):** a *critical-path control that is fundamentally AT-SPI-invisible* — a `<canvas>`-drawn model picker, or a Wayland-portal chooser on the critical path. Here match-or-notify has **no ground truth to match**. A "fallback" to OCR/vision is genuinely probabilistic (confidence thresholds = fuzzy), so the contract's purity correctly forbids it. The honest resolution is **scope exclusion**, not a new branch: the contract must, per platform, *verify and assert that every critical-path control is AT-SPI-visible* as a precondition. If any isn't, that platform is declared **"not 100%-deterministic via AT-SPI"** — out of scope, or vision-assisted with the explicit acknowledgment that this **leaves the deterministic model**. I'd argue no fallback is justified *inside* the deterministic surfaces; the only place purity costs you coverage is exactly here, and the contract should own it rather than imply completeness. (CAPTCHA, by contrast, is not a fallback case — it's precisely notify+halt-for-human, which the model already endorses.)

## (d) Stickiness — the real deviation channels

The enforcement (L839–844) protects **code**, but the contract just **moved all the variability into the YAML** — and into the shared primitives — which the code-lint and the driver-merge-gate may not cover. Four ways a future operator/editor still deviates:

1. **Hand-edit the YAML to a looser map** (drop the states/scope, broaden a name). That's a *fuzzy match in data*; a code-grep never sees it. **Fix:** YAML maps are part of the risky surface — same `audit/grok` + `audit/gatekeeper` repro gate as drivers, **explicitly**. Add a YAML **schema lint** (every element has role + required-states + unique scope-path; no substring operators in the YAML either). Version-stamp the YAML; each run records the version it ran against (provenance).

2. **Bury a fallback in a shared primitive** — make `click` retry-on-miss or `snapshot` "best-effort." Centralizes the deviation in "infra" the driver-targeted gate may exempt. **Fix:** primitives are in-scope for the lint and the gate; and require interaction primitives to emit events the framework actually consumes (your `click-react` exists for this reason — generalize it as a *fidelity invariant*, else clicks register in the a11y tree but the app never reacts → silent "clicked, no transition").

3. **The lint is syntactic; swallowing can be semantic** — a `try/except` that logs-and-continues, or `return matches[0] if matches else SENTINEL` treated as proceed. No `name_contains` token to grep. **Fix:** make the property **type-enforced and behaviorally-tested**, not grep-enforced. Every validate returns an explicit `Match | NoMatch` sum type the caller must handle exhaustively; add a behavioral gate that **injects a miss and asserts a notify fires**. Grep-lint is necessary, not sufficient.

4. **Box-hotfix bypasses the merge gate entirely** — "prod's down, 5 consultations stuck, just edit the YAML/driver on the running machine." It never merged, so grok/gatekeeper never see it. This is the channel your current enforcement *cannot* see. **Fix:** the **running engine refuses to load un-gated artifacts** — sign the YAML+driver bundle, verify signature at load, unsigned/locally-modified → refuse (or run loud-"UNVERIFIED" that itself notifies). Without this, the entire gate is advisory the moment someone has SSH.

And the soft underbelly beneath all four: **"this file is canonical… stale lists are subordinate" (L844) is a claim, not a mechanism.** Nothing *executable* forces the contract to win over code+YAML at runtime; the link is human discipline + audit. **Strongest fix:** make the contract and what-runs the *same artifact* — emit a machine-readable appendix (the per-step state sets, the state machine, the disposition/reason-code table) that the engine **loads**, so "contract" and "behavior" can't drift without the loader noticing. As long as they're two things kept in sync by audit, contract↔code drift is itself an un-gated risk.

---

**Verdict:** Not yet airtight — the *model* (match-or-notify, no-guess) is sound, but the *contract's "match" is under-specified*: it assumes one expected element per step located by visible `name` and treats every miss as drift, so alternate states (auth/rate-limit/error/partial/modal), duplicate/disabled elements, dynamic names, and wrong-window matches slip through as false-drift or false-success. **Single most important fix:** redefine "match" = *exactly one* node by **{stable-locator + role + required-states}** in a *mapped scope*, and "step match" = *exactly one of an enumerated set of mapped states* (happy + every reachable alternate), each with a deterministic disposition — then put the YAML and primitives under the same signed, repro-gated merge path as the drivers.
