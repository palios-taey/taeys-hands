## What Was Built

Comprehensive three-layer session state synchronization to prevent browser/MCP/database mismatches.

### The Problem

Before this implementation, the system had THREE disconnected layers:

1. **Browser Session** (Playwright) - Actual browser tab state
2. **MCP Session** (SessionManager) - Runtime session registry
3. **Database Session** (Neo4j) - Persistent session records

These layers **never synchronized**, causing:
- Orphaned sessions after server restart
- No detection of browser crashes
- State drift (browser at URL X, database says URL Y)
- Lost conversation context after compacts
- Confusion between sessionId and conversationId

### The Solution

Four new synchronization mechanisms:

1. **updateSessionState()** - Syncs browser URL to database automatically
2. **getSessionHealth()** - Validates session integrity and detects problems
3. **reconcileOrphanedSessions()** - Detects and marks orphaned sessions on startup
4. **Platform registry** - Single source of truth for conversation URL patterns

## How It Works

### Terminology Clarity

**sessionId** = MCP session UUID (ephemeral, generated by SessionManager, lost on restart)
**conversationId** = Platform chat ID (persistent, extracted from browser URL, survives restarts)
**conversationUrl** = Full browser URL (e.g., `https://claude.ai/chat/abc-123-def`)

### Synchronization Flow

Every tool call should follow this pattern:

```typescript
// 1. PRE-FLIGHT: Validate session health
await sessionManager.validateSessionHealth(sessionId);

// 2. EXECUTE: Perform browser operation
const result = await chatInterface.someOperation();

// 3. POST-SYNC: Update database with current state
const currentUrl = await chatInterface.getCurrentConversationUrl();
await conversationStore.updateSessionState(sessionId, currentUrl, platform);
```

This ensures all three layers (Browser, MCP, Database) stay synchronized.

### Orphan Recovery

On server startup:

```typescript
// Detect orphaned sessions from previous run
const sessionManager = getSessionManager();
await sessionManager.syncWithDatabase(conversationStore);
```

This marks all 'active' sessions as 'orphaned' (since MCP session registry is empty after restart).

Users can then:
- List orphaned sessions
- Resume with `taey_connect({ conversationId })`
- Or ignore and start fresh

### Before vs After

| Scenario | Before | After |
|----------|--------|-------|
| **Server restart** | All sessions lost, no way to reconnect | Orphaned sessions detected, can resume with conversationId |
| **Browser crash** | Cryptic Playwright errors | Clear error: "Session dead (browser crashed)", can reconnect |

- ✅ MCP server restarts (sessions marked orphaned, recoverable)

- ✅ State drift (automatic sync prevents divergence)
- ✅ Compacts (no more lost context!)

## Example Usage

### Complete Tool Handler

```typescript
case "taey_send_message": {
  const { sessionId, message, waitForResponse } = args;

  try {
    // 1. Pre-flight health check
    await sessionManager.validateSessionHealth(sessionId);

    // 2. Get interface and platform
    const session = sessionManager.getSession(sessionId);
    const chatInterface = session.interface;
    const platform = session.interfaceType;

    // 3. Execute operation
    await chatInterface.sendMessage(message, { waitForResponse });

    // 4. Sync state to database
    const currentUrl = await chatInterface.getCurrentConversationUrl();
    const { conversationId } = await conversationStore.updateSessionState(
      sessionId,
      currentUrl,
      platform
    );

    // 5. Return success
    return {
      success: true,
      sessionId,
      conversationId,
      message: 'Message sent successfully'
    };
  } catch (err) {
    // Clear error message if session dead
    if (err.message.includes('dead')) {
      return {
        success: false,
        error: 'Session is dead. Please reconnect with taey_connect.',
        sessionId
      };
    }
    throw err;
  }
}
```
